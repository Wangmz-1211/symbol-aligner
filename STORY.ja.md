# Symbol Aligner — プロジェクトの背景

## このプロジェクトが生まれた経緯

あるレガシープロジェクトのリファクタリングの過程で、少々厄介な状況に陥りました。リファクタリングが業務上の命名規約の変更を伴っていたため、大量のコードをリネームする必要が生じたのです。プロジェクトには非標準的な命名のコードが大量に存在していたため、プロジェクト全体を単純に一括置換すると、誤置換や置換漏れが多発します。一方で手作業による修正は、非常に時間がかかるうえ、品質を一定に保つことも困難でした。

まず、この問題がなぜ難しく、なぜ重要なのかを説明します。

命名リファクタリングには次のような特徴があります。

1. テキストの大半は複合語であり、通常はいくつかの単純なテキストマッピング（以下「サブマッピング」と呼ぶ）から構成される。
2. サブマッピングはすべての複合語に適用できるわけではない。2 つのサブマッピングからなる複合マッピングは単純な連結では**なく**、一方のサブマッピングを無効化してしまう場合があり、しかもそこに法則性はない（これは業務の設計に依存するもので、技術チームの意図ではない）。
3. 初期開発者の悪習により、コードには非標準的な命名が存在するが、それらは標準的な命名と高い類似度を持つ。母音の省略やスペルミスなどである。

さらに、このコードベースはアーキテクチャ設計が混乱し、モジュール間の結合度が高いという問題を抱えており、これがリネームの難易度を大幅に押し上げていました。変数のスコープが単一ファイルの範囲を超えると、信頼性の低い変換はプロジェクト全体により多くのコンパイルエラーを持ち込みます。ファイル間で同期の取れない変換はインデックス構築も壊し、LSP は変換前と変換後の変数を別個の要素として扱いがちで、コンテキストの一貫性を破壊します。

そのため、大規模なコード変換に着手する前に、まず命名変換問題に対する**高信頼性**の手法を見つけることが不可欠でした。さもなければ、後続の作業に与える悪影響は計り知れません。

## チームでの初期の試み

プロジェクトの初期段階では、メンバーから多くの解決策が提案されましたが、実践の結果、それらは効率が悪いか、精度要件を満たせないかのいずれかでした。本リポジトリの設計を紹介する前に、私が非常に非効率だと考えるこれらの初期案を振り返り、その不合理な点を分析しておく価値があります。

### 案 1

最も早い案は、私がプロジェクトに参加する前に既に提案されていたものです。

この手法は単純なマッピング関係とブラックリスト機構に基づいていました。すなわち、存在しうるすべてのサブマッピング項目と、サブマッピングが存在しても変換を禁止する項目を列挙し、それをプロンプトの一部として Agent に与えます。Agent は独立したセッションで単一ファイルを処理します。この案は複数のプロンプトファイルを伴い、2 つのステップに分かれていました。

1. 人間が事前に用意したプロンプトファイルと対象ファイルを提供し、Agent が指示に従ってファイル内を検索・分析し、リネームが必要なすべての箇所を、あらかじめ定義された形式の Markdown テーブルにまとめる。確信度の低いマッピングについては、人間が唯一の信頼できる情報源（共有スプレッドシート）を照会し、あらかじめ定義された方法で Agent に返信する必要がある。
2. ステップ 1 の返信によって Agent がマッピング関係を把握したのち、別の事前準備済みプロンプトを与え、Agent があらかじめ定義されたフローに従って修正を行い、最終的にあらかじめ定義された形式で出力する。

振り返ると、いくつかの問題点が浮かび上がります。

1. 明らかに、プロセス全体が固定的でした。固定のサブマッピング、固定のワークフロー、固定の出力形式。にもかかわらず、この振る舞いは再利用可能な Skill として結晶化されませんでした。
2. 提案者たちは、最初に述べた命名変換の特徴 2、すなわち複合語のマッピングがサブマッピングの単純な積み重ねではないことに気づいていなかったわけではありません——だからこそワークフローに人手によるレビュー工程を加えたのです。しかしそれでもこの問題を一時的に無視することを選び、技術的負債を人的コストへと転嫁しました。
3. この案は計算コストを考慮していませんでした——2026 年 6 月以前の GitHub Copilot の課金体系では大きな問題ではありませんでしたが、現在の体系下では算力消費は考慮せざるを得ない問題です。

この案は、Agent が安定したマッピングを持つツールではなく、確率モデルの上に構築された知能を持つ産物である——だからこそ *AI* Agent と呼ばれる——という点を見落としていました。最終的に、実践はこの案が精度要件を満たせないことを示し、この手法で変換されたファイルは後続の変換・テスト工程にいくつかの潜在的リスクを残しました。

### 案 2

案 1 が頓挫したのち、別のチームが新たな案を提案しました——おそらく MAS 設計と AI ハーネスの思想に影響を受けたのでしょう、彼らはこの命名変換問題を扱う Skill を開発しました。複雑な MAS を設計し、Orchestrator で全体のスケジューリングを行い、複数のロールを定義してかなり長いワークフローを遂行させました。記憶では少なくとも 5 つの段階があり、各段階にはさらに数種のバリエーションがあり、可監査性を高めるために各段階の成果物をレポートで管理していました。

最終的に、この案は次のような特徴を示しました。

1. Agent の推論能力に過度に依存していたため、命名変換の品質を保証できなかった。
2. フローが長いため、微小な誤差がフローの中で大きな誤りに積み重なりうる。
3. フローが長いため、算力の消費が非常に大きい。これは現在の GitHub Copilot の課金方式の下では予算のブラックホールとなった。
4. フローが長いため、各段階がレポートを出力するとはいえ、問題発生時に素早く誤りを特定できるわけではなく、プロセス自体を反復改善する助けにはほとんどならなかった。
5. フローが長いため、実行時間が長すぎて効率が悪い。

以上を総括すると、2 つの案は地続きであり、共通の核心的な発想を持っています——仕事をそのまま Agent に丸投げすれば、何でもやってくれる、というものです。案 2 は案 1 に対して大きく改善されており、MAS を用いてコンテキストを管理し、可監査性と、制御可能で再利用可能なプロセス設計に注力しています。しかしながら、非常に基本的な誤りを犯しました。誤差の増幅効果を無視し、人間でさえ理解しがたいほど複雑なプロセスを設計してしまったのです。

## 本手法

個人的には、ここ数か月で LLM の能力が急速に向上するなか、多くのエンジニア（私自身を含む）が Agent の能力の境界について必ずしも正確でない認識を抱いていると考えています。複雑な問題に対して、Agent はさまざまな解決策を提案しうりますが、それらが信頼でき、効率的で、実現可能かどうかを保証する者は誰もいません。したがって、この種のアプリケーションを開発する際には、**AI は信頼できない**と常に意識することが基本的な方向性となります。

さらに、多くの人は比較的高レベルな Agent にしか触れられず、その基盤となる LLM には不慣れなようです。一部の単純な問題に対しては、CPU 上に 7B パラメータのモデルをデプロイすれば核心を解決でき、残りの部分はすべてスクリプトとして書けるかもしれません。しかし多くの人は、複雑な問題全体を Agent で処理することしか思いつきません。彼らの努力の方向は、最初から間違っていたのです。

本プロジェクトに存在する主要な問題に対して、本プロジェクトでは次のように対処しています。

| 問題 | 解決策 |
| --- | --- |
| サブマッピングの非線形な積み重ね | 複合マッピングは有限の集合であり、完全なマッピング集合を維持する必要がある |
| 非標準的な命名 | 複数のファジーマッチングアルゴリズムの加重を用い、多様な誤りをカバーする |
| 人手レビューの削減 | LLM はマッチ結果の判定にのみ用い、マッピング集合内の結果だけを対象とする |
| キーワードが異なる識別子に埋め込まれうる | AST で解析し、識別子の種類ごとに異なる方法でマッチする |

前述の 2 つの案のその他の不足点について、本手法では次のように解決しています。

1. **再利用性**：本手法はスクリプトとして実装し、MCP ツールとして公開する。
2. **計算コスト**：本手法は LLM のみを使用し、理論上のコストは Agent の数十分の一（あるいは百分の一）にすぎない。
3. **Agent の推論への過度な依存**：本手法の大部分は伝統的なマッチングアルゴリズムであり、LLM はファジーマッチング時の意味的リコールにのみ使用する。
4. **可監査性**：リネームはスクリプトと AST に基づくため、監査データの準備が簡潔かつ信頼できる。

詳細な設計は [README](./README.ja.md) を参照してください。
